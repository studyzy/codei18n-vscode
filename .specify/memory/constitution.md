# Codei18n VSCode Extension 项目章程

## 核心原则

### I. 包装器架构 (Wrapper Architecture)
本项目的核心职责是将 `../codei18n` 的功能封装为 VSCode 插件。
*   **核心逻辑分离**: 所有的核心业务逻辑、国际化处理算法必须维护在 `../codei18n` 项目中，而不是本插件中。
*   **依赖引用**: 本插件应通过适当的方式（如子模块、库引用或 CLI 调用）使用 `codei18n` 的功能。
*   **职责边界**: 本插件仅负责 VSCode 界面交互、配置管理、命令注册以及与编辑器的集成。

### II. VSCode 原生集成 (VSCode Native Integration)
必须遵循 VSCode 插件开发的最佳实践和规范。
*   **API 使用**: 优先使用 VSCode 提供的官方 API，避免使用非公开或不稳定的 API。
*   **用户体验**: 遵循 VSCode 的设计语言（例如通知、状态栏、快捷键、配置项命名）。
*   **性能**: 确保插件的加载和执行不会阻塞 VSCode 的主线程，耗时操作必须异步执行。

### III. 文档与注释 (Documentation & Comments)
代码的可维护性和可读性至关重要。
*   **代码注释**: 所有导出的函数、类、接口必须包含清晰的注释（JSDoc），说明其用途、参数和返回值。
*   **复杂逻辑**: 对于复杂的逻辑块，必须在代码内部提供解释性注释。
*   **README**: 项目根目录必须包含详细的 README，说明如何安装、配置和使用插件，以及开发指南。

### IV. 单元测试覆盖 (Unit Testing Coverage)
必须编写足够的单元测试以保证代码质量。
*   **测试优先**: 鼓励采用测试驱动开发 (TDD) 的思维方式。
*   **覆盖率**: 核心功能模块必须有相应的单元测试覆盖。
*   **测试框架**: 使用 VSCode 官方推荐的测试框架（如 `vscode-test`, `mocha`）。
*   **CI 集成**: 测试应能够自动化运行，确保每次提交不破坏现有功能。

### V. 版本控制 (Versioning)
采用语义化版本控制 (Semantic Versioning)。
*   **格式**: MAJOR.MINOR.PATCH (例如 1.0.0)。
*   **主版本 (MAJOR)**: 当进行不兼容的 API 修改或重大架构变更时增加。
*   **次版本 (MINOR)**: 当添加向下兼容的功能时增加。
*   **补丁版本 (PATCH)**: 当进行向下兼容的问题修正时增加。

## 治理

本项目章程是项目开发的最高指导原则。
*   **修改流程**: 任何对章程的修改都需要经过团队讨论和批准。
*   **版本更新**: 每次修改章程必须更新版本号和修订日期。
*   **合规性**: 所有提交的代码 (PR) 都必须符合本章程定义的原则。
